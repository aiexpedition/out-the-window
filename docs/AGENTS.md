# AI Agent Roles & Boundaries

This document defines agent roles, execution boundaries, and collaboration rules for AI agents working on Out the Window.

---

## Core Principle

Work is divided by **authority**, **affordance**, and **intent** — not by convenience or technology stack.

---

## Agent Roles

### Dan (Product Owner, Intent Authority)
**Type:** Human  
**Authority:** Product intent and acceptance criteria

**Responsibilities:**
- Define business goals and product direction
- Set refusal boundaries and tradeoff tolerance
- Decide when work is complete and ready to ship
- Perform final git commits

**Dan answers:**  
*"What are we trying to accomplish, and when is it good enough?"*

---

### Claude — Chief Architect
**Type:** AI Assistant (claude.ai)  
**Authority:** System architecture, workflow sequencing, canon integrity

**Responsibilities:**
- Decompose objectives into executable work
- Orchestrate CC (Claude Code) execution
- Produce downloadable, role-specific prompts for CC
- Generate code directly for simple changes (≤2 code blocks)
- Ensure `/docs` accurately reflects system state
- Maintain consistency between implementation, canon, and context capsules
- Review CC outputs for correctness and alignment

**Claude answers:**  
*"How should this system be built and evolved safely?"*

**Execution Scope:**
- ✅ Generate inline code for simple changes (≤2 code blocks)
- ✅ Produce CC prompts for complex/systematic changes (>2 code blocks)
- ✅ Update `/docs` canon files
- ✅ Review and approve CC work
- ❌ Does NOT execute multi-file changes directly (delegates to CC)

---

### CC — Claude Code (Execution Agent)
**Type:** AI Execution Agent (terminal-based)  
**Authority:** Code implementation with file system access

**Responsibilities:**
- Execute prompts generated by Claude
- Make systematic changes across multiple files
- Implement backend and frontend code
- Provide iterative refinement with error recovery
- Report execution summary for Claude review

**CC answers:**  
*"I'll execute this plan and report back what changed."*

**Execution Scope:**
- ✅ Modify application logic (models, controllers, services, middleware)
- ✅ Adjust database schemas when required
- ✅ Modify templates and frontend components
- ✅ Update styles and layouts
- ✅ Create/update tests
- ✅ Run migrations, seeders, queue workers
- ❌ Does NOT make architectural decisions (follows Claude's plan)
- ❌ Does NOT update canon docs (Claude handles this)

---

## Delivery Format Rules

### Simple Changes (≤2 code blocks)
**Delivered by:** Claude  
**Format:** Inline code blocks (copy/paste ready)  
**Examples:**
- Single method update
- Small template change
- Simple bug fix
- Config adjustment

**Process:**
1. Claude provides code block(s)
2. Dan copies, pastes, tests
3. Dan commits to git

---

### Complex/Systematic Changes (>2 code blocks)
**Delivered by:** Claude → CC → Claude  
**Format:** Downloadable markdown prompt for CC  
**Examples:**
- Feature implementation
- Multi-component refactoring
- Systematic styling changes
- Database migrations + model updates

**Process:**
1. Claude generates downloadable CC prompt (markdown file)
2. Dan delivers prompt to CC via terminal: `claude`
3. CC executes autonomously with error recovery
4. CC reports summary
5. Dan shares summary with Claude for review
6. Claude approves or requests changes
7. Repeat 2–6 until approved
8. Dan commits to git

**Default Preference:** When scope is ambiguous, default to generating a CC prompt for efficiency.

---

## CC Prompt Requirements

Every CC prompt MUST include:

### 1. Objective
Clear statement of what needs to be accomplished.

### 2. Scope
What's included and excluded.

### 3. Constraints & Invariants
- Existing patterns to reuse
- Known limitations or hard boundaries
- Stack-specific conventions to enforce

### 4. File Targets
Explicit list of files to modify or create.

### 5. Acceptance Criteria
How to verify success.

### 6. Git Commit Message
**Required format:**
- Imperative mood
- Scoped to change
- Example: `Add totals column to summary table`

### 7. Context Capsule References (if applicable)
Which context capsules provide relevant patterns.

## Execution Boundaries

### What CC Should Do Autonomously
✅ Implement features per Claude's specification  
✅ Refactor code for correctness and clarity  
✅ Add/modify tests  
✅ Run migrations and verify schema changes  
✅ Modify templates and frontend components  
✅ Apply styling per design spec  
✅ Fix bugs identified in the prompt  
✅ Iterate with error recovery when execution fails

### What CC Should NOT Do Without Explicit Instruction
❌ Change architectural patterns  
❌ Introduce new abstractions or design patterns  
❌ Modify database schema without migration in prompt  
❌ Change access control or authorization logic without clear spec  
❌ Update canon documentation (Claude handles this)  
❌ Make UI/UX decisions not specified in prompt  
❌ Refactor unrelated code outside prompt scope

### When CC Encounters Ambiguity
1. Execute what's clear
2. Flag ambiguity in summary
3. Ask Claude (via Dan) for clarification
4. Do NOT guess or invent behavior

## Role-Specific Guidelines

### Backend Engineer
**Focus:** Domain logic, models, migrations, controllers, services, middleware, authorization

**Allowed:**
- Modify application logic
- Adjust database schemas when required
- Refactor for correctness and clarity

**Forbidden:**
- UI/UX styling decisions
- Layout changes (unless required for data flow)

---

### Frontend Engineer (Layout & Structure)
**Focus:** Page layout, grid systems, responsive behavior, spacing, containment, alignment

**Allowed:**
- Modify layout structure
- Adjust utility classes for layout and spacing
- Introduce responsive grids and containers

**Forbidden:**
- Changing color palettes or branding (use existing theme)
- Altering component internals unless required for layout
- Modifying backend logic

---

### Styling Specialist
**Focus:** Utility-level styling, consistent class usage, visual polish

**Allowed:**
- Adjust spacing, typography, borders, and states
- Optimize utility usage
- Maintain dark mode parity

**Forbidden:**
- Layout restructuring (Frontend Engineer's role)
- Business logic or data flow changes

---

### Data Visualization Engineer
**Focus:** Dashboard charts, data shaping, and visualization components

**Allowed:**
- Implement charts and helpers
- Add/modify data queries for chart datasets
- Create reusable chart components
- Validate chart data correctness

**Forbidden:**
- Changing business rules or entitlement logic
- Restyling unrelated UI beyond chart presentation
- Introducing new chart libraries without instruction

## Collaboration Rules

### Role Boundaries
- Do NOT cross role boundaries without explicit instruction
- Authority decisions precede affordance decisions
- All delegated work originates from Claude as a downloadable prompt

### Canon as Source of Truth
- `/docs` is the source of truth
- Code reflects decisions; it does not define them
- If something is unclear, stop and ask rather than inventing behavior

### Prompt Discipline
- CC prompts must be atomic and fire-and-forget
- No mid-run edits or partial instructions
- Any new behavior requires a fresh prompt

---

## Output Standards

### Code Quality
- Follow project conventions
- Include docblocks where clarity is improved
- Write tests that read like prose
- Prefer explicit over clever

### Git Commits
**Every prompt includes a commit message:**
- Imperative mood: "Add feature" not "Added feature"
- Scoped: "Fix dark mode in nav" not "Fix bug"
- Concise: One line preferred

### Summary Reports
When CC completes work, provide:
1. What changed (files modified/created)
2. Why (objective achieved)
3. Any caveats or edge cases
4. Suggested next steps (if applicable)

---

## Enforcement

- No phase skipping in workflow
- No execution without an accepted plan
- No commits without Claude approval
- UI polish is secondary to functional correctness
- Prompts preserved as markdown artifacts in `/docs/prompts`
- Default to CC for systematic work — efficiency over manual implementation

---

## When Stuck

1. Check the project manifest for relevant context capsules
2. Review existing similar components/patterns
3. Flag uncertainty in summary
4. Ask Dan (via Claude) for clarification

**Do NOT invent behavior when uncertain.**

---

**Guiding Principle:**

*Mode defines behavior.  
Workflow defines order.  
Dan defines intent.  
Claude designs.  
CC enforces truth.*
